# glBitmap function


## -description


The <b>glBitmap</b> function draws a bitmap.


## -parameters




### -param width

The pixel width of the bitmap image.


### -param height

The pixel height of the bitmap image.


### -param xorig

The <i>x</i> location of the origin in the bitmap image. The origin is measured from the lower-left corner of the bitmap, with right and up directions being the positive axes.


### -param yorig

The <i>y</i> location of the origin in the bitmap image. The origin is measured from the lower-left corner of the bitmap, with right and up directions being the positive axes.


### -param xmove

The <i>x</i> offset to be added to the current raster position after the bitmap is drawn.


### -param ymove

The <i>y</i> offset to be added to the current raster position after the bitmap is drawn.


### -param bitmap

The address of the bitmap image.


## -returns



This function does not return a value.




## -remarks



A bitmap is a binary image. When drawn, the bitmap is positioned relative to the current raster position, and framebuffer pixels corresponding to 1s in the bitmap are written using the current raster color or index. Frame-buffer pixels corresponding to zeros in the bitmap are not modified.

The bitmap image is interpreted like image data for the <a href="https://msdn.microsoft.com/c4eda64f-a75e-4e6d-984d-af49414b94d8">glDrawPixels</a> function, with <i>width</i> and <i>height</i> corresponding to the width and height arguments of that function, and with <i>type</i> set to GL_BITMAP and <i>format</i> set to GL_COLOR_INDEX. Modes you specify using <a href="https://msdn.microsoft.com/c12255eb-8802-4602-b1fa-ab157201ae5c">glPixelStore</a> affect the interpretation of bitmap image data; modes you specify using <a href="https://msdn.microsoft.com/c14349c0-ff50-441f-b9fd-8b0f5114fd8a">glPixelTransfer</a> do not.

If the current raster position is invalid, <b>glBitmap</b> is ignored. Otherwise, the lower-left corner of the bitmap image is positioned at the following window coordinates:

<i>x</i><sub>w</sub> = <i>x</i><sub>r</sub> <i>x</i>ₒ

<i>y</i><sub>w</sub> = <i>y</i><sub>r</sub> <i>y</i>ₒ

In these coordinates, (<i>x</i><sub>r</sub> , <i>y</i><sub>r</sub> ) is the raster position, and (<i>x</i>ₒ , <i>y</i>ₒ ) is the bitmap origin. Fragments are then generated for each pixel corresponding to a 1 in the bitmap image. These fragments are generated using the current raster <i>z</i>-coordinate, color or color index, and current raster texture coordinates. They are then treated just as if they had been generated by a point, line, or polygon, including texture mapping, fogging, and all per-fragment operations such as alpha and depth testing.

After the bitmap has been drawn, the <i>x</i> and <i>y</i> coordinates of the current raster position are offset by <i>xmove</i> and <i>ymove</i>. No change is made to the <i>z</i>-coordinate of the current raster position, or to the current raster color, index, or texture coordinates.
